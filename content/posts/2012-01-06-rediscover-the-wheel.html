--- title: Learning to Rediscover the wheel ---

I didn’t like Jekyll. So one afternoon I hacked together a static-site generator called Fairytale. Comparatively it’s not as capable or well designed. Yet I can now appreciate why Jekyll works as it does despite my initial dislike.

This is an example of something of a personal rule: if you don’t understand something then build a simpler way to solve the problem. Crucially this isn't the same as building a clone: look for a sensible way to do things, not just the one that they chose.

In a similar way to how the creators of something are often amongst the greatest experts on using it, those who’ve fought with a problem are better placed to think intelligently about a solution.

In some ways this approach doesn’t help get stuff get done. After all it amounts to the opposite of “don’t reinvent the wheel”. Yet I've always felt that if trying to learn something new you can’t become an expert by just using what’s already been done. Using code libraries is a necessity for a productive programmer, yet you don’t learn much from it.

The more you do this the more you widen your knowledge. This time I’ve ended up using HAML & RAM Disk just because they seemed interesting. From ORMs (“use only if necessary”) to personal hashing algorithms (“interesting but maybe not a good idea”) I’ve ended up reaching a lot of personal turning points just by doing ‘impractical’ things.

As for Fairytale? You're reading the output right now, served from RAM disk.
